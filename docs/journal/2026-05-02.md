### agent-browser 復活ループの罠

**現象**:
1. `agent-browser close --all` → CDP セッション切断、応答 "✓ Closed session: default"
2. `pkill -9 -f "agent-browser-chrome"` → chrome 群を SIGKILL
3. **数秒後に新 chrome 群が自動再起動**（PID 帯が新しい、user-data-dir UUID も別物）
4. ループ繰り返し → メモリ解放されず

**真犯人の特定**:
```bash
ps -eo pid,ppid,etime,rss,comm | grep agent-browser-darwin
# → 44314  1  00:38  10192  agent-browser-darwin-arm64
```
parent PID 1 の **孤立デーモン** `agent-browser-darwin-arm64` が裏で生き続け、chrome を自動 spawn し直していた。`close --all` は CDP 接続を閉じるだけで supervisor 自体は止めない仕様。

**完全停止コマンド**:
```bash
pkill -9 -f "agent-browser-darwin"   # supervisor を先に殺す
pkill -9 -f "Chrome for Testing"     # chrome 子プロセス群
pkill -9 -f "agent-browser-chrome"   # 念押し
```
3 コマンド連射で復活ループ停止。

### Codex config.toml クリーンアップ

**問題**:
1. `[mcp_servers.peekaboo]` セクション = peekaboo は Mac 専用ツール、Windows で起動失敗
2. `mcp-search` = claude-mem プラグイン由来（thedotmack/claude-mem）、bun 未導入で起動失敗

**Codex の MCP startup ログ**:
```
MCP client for 'peekaboo' failed to start: MCP startup failed: program not found
MCP client for 'mcp-search' failed to start: MCP startup failed: program not found
MCP startup incomplete (failed: mcp-search, peekaboo)
```

**手順**:
1. `Copy-Item config.toml config.toml.bak-20260502` でバックアップ
2. **正規表現の罠**: 最初 `^\[mcp_servers\.peekaboo\][^\[]*` で削除しようとしたが、section 内の `args = ["mcp"]` の `[` で early termination → `["mcp"]` と `command = "peekaboo"` が残骸化、設定壊れる
3. **修正**: バックアップから復元 → line-based filter（`StartsWith('[')` で section header 検出、次の section header で skip 終了）
4. claude-mem plugin の `enabled = true` → `enabled = false`（PowerShell `-replace` で置換）
5. UTF-8 BOM なしで `[System.IO.File]::WriteAllText` 経由 save

**結果**:
- `[mcp_servers]` セクションには `playwright` のみ残存
- `[plugins."claude-mem@thedotmack"]` `enabled = false`
- 次回 Codex 起動時に warning 消える想定

**学び**:
- TOML 編集時、regex `[^\[]*` は **value 内の `[`** で early termination する罠 → line-based filter 推奨
- PowerShell -replace は dollar sign や バックチック escape 必要、ASCII-only スクリプトが安全
- Windows PowerShell 5 のデフォルト出力エンコードは Shift_JIS、SCP で送ったファイルは UTF-8 でも正しい（Get-Content 表示が文字化けしてもファイル本体は無事）
- 検証は roundtrip diff（送ったファイルを scp で取得し直して `diff` で確認）

## 追補 6: ssh config を mDNS 動的解決に変更（07:47）

### 動機

しゅん先生 PC は再起動ごとに DHCP で IP が変動する。Tailscale 不採用方針なので IP は固定化されない。**手動で ssh config 更新は摩擦が大きい**。

### 解決: mDNS hostname (`.local`) で都度解決

`~/.ssh/config`:

```
Host shun-sensei
    HostName DESKTOP-ATQ36KS.local
    User wirel
    IdentityFile ~/.ssh/id_ed25519
    StrictHostKeyChecking accept-new
```

macOS の mDNSResponder が `.local` 名を自動で IP に解決。再起動で IP が変わっても ssh コマンドはそのまま `ssh shun-sensei`。

### 検証

```
$ ssh shun-sensei 'powershell -Command "hostname; whoami; (Get-Date).ToString(\"yyyy-MM-dd HH:mm:ss\")"'
Warning: Permanently added 'desktop-atq36ks.local' (ED25519) to the list of known hosts.
DESKTOP-ATQ36KS
desktop-atq36ks\wirel
2026-05-02 07:47:19
```

→ `ssh shun-sensei` 一発で接続成立。

### 関連 known_hosts

- `desktop-atq36ks.local` の host key を accept-new で初登録
- 旧 IP ベースのエントリ（`192.168.2.187`, `192.168.2.56`, `192.168.2.248`）は残存だが影響なし、host key 同じ
- 将来の cleanup TODO: 同 host key の重複エントリを統合可能（必須ではない）

### 制約条件（前提）

- **同 Wi-Fi にいる時のみ動作**（mDNS は L2 ブロードキャスト依存）
- LAN 外（自宅・別コワーキング）からは masu-p55 (HP ProBook、Tailscale 100.125.21.47) 経由 SSH ホップ必要
- mDNSResponder が動いてないと解決失敗（macOS 通常起動時はデフォルト稼働）
