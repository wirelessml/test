### BT ROUND キーボード (KOR-05) ペアリング

- 製品: コーリュウ株式会社「丸型ワイヤレスキーボード」、Bluetooth 5.0 / HID プロファイル
- 単 4 × 2 本 (別売)、連続動作 80 時間、84 キー、約 33 × 14 × 2.6 cm、ABS、技適 210-209936
- ペアリング手順:
  1. Fn + E で **Windows モード**に切替 (LED 3 回点滅)
  2. Fn + C を 3 秒長押し → ペアリングモード (LED 点滅)
  3. しゅん先生 PC 側「設定 → Bluetooth → デバイス追加」→ 「BT ROUND」選択
  4. PIN 求められれば 0000 (今回不要だった)
- 接続確認: しゅん先生 PC の `Get-PnpDevice -Class Bluetooth` に **BT ROUND OK** 表示
- 注意: 一度に 1 台のみ、電源 ON/OFF なし (使わない時は電池抜く)、10 分無操作スリープ

### 🚨 4/30 午後の「Google日本語入力切替不可」結論を完全撤回

#### きっかけ
LIAM (@liappleq_1283) のツイート「俺が MacBook 買って 1 番最初に入れるものこれ」(4/30 19:09、38 ❤️ / 1,096 views) に Google日本語入力の画像。4/30 午後の自分の検証結果との矛盾を確認すべく Web 検索。

#### Web 検索で判明
- Google公式 Gboard コミュニティに「Google日本語入力が macOS 26 で正常に動作しません」スレッド存在 (内容は WebFetch でナビ枠のみ取得)
- 北村事務所ブログ等で「**インストール後に Mac を再起動**」と明記
- v3.33 から表示名は「**ひらがな (Google)**」(かっこ内表記、4/30 検証で「Google」検索ヒット 0 だった理由)

#### 動作確認 (5/1 11:45 頃)
- Mac 再起動後、System Settings → キーボード → 入力ソース → 「+」→ 日本語 → **「ひらがな (Google)」** 追加 → ✓ 動作
- メニューバーに「あ ひらがな (Google)」アクティブ、ショートカット ⌃⇧J/K/L/; 等利用可
- Google IME メニュー (再変換 / 環境設定 / 単語登録 / 辞書ツール) 全機能アクセス可
- Apple ことえり (`日本語 - ローマ字入力`) を「-」で削除、卒業完了

#### 真因と訂正

| 4/30 午後の結論 (誤) | 5/1 朝の事実 (正) |
|---|---|
| 「macOS 26.5 Tahoe + Google IME 3.33 は三重ブロックで不可能」 | **再起動さえすれば普通に動く** |
| System Settings UI に「Google」が出ない | 再起動前の未登録状態だった、表示名は「ひらがな (Google)」 |
| TIS API `-50 paramErr` でブロック | 未登録状態への正しいエラー応答だった可能性 |
| defaults plist 直書きを OS が即 revert | 入力ソースとして登録されてないので OS が拒否したのが正しい |

→ **真因は「再起動を試さなかった」一点**。三重ブロックは全部「再起動前の未登録状態を観察していた」だけ。journal 4/30 追補 1.4 の「Google日本語入力切替不可」結論は撤回。

#### 学んだこと
- macOS の入力ソース登録は **ファイル配置 + 再起動 = 反映** という古典的パターン
- v3.33 で表示名が「Google日本語入力」→「ひらがな (Google)」等に変わった (4 モード並列表示)
- 「不可能」の結論を出す前に、**OS のお決まりの儀式 (再起動)** を試すべきだった
- Web 検索で公式コミュニティ + サードパーティブログを 1 つでも当たっていれば最初から判明した問題

### 学んだこと

- **synthetic KeyboardEvent dispatch は React app に届く**: `window.dispatchEvent(new KeyboardEvent(...))` で OK、`isTrusted: false` でも処理されるアプリは多い
- **Playwright の `keyboard.type` が届かないケースがある**: agent-browser/Playwright 経由の CDP keyboard event が React の window-level listener に届かない原因は未解明 (フォーカス管理 or イベントターゲット差?)、要追跡
- **ブラウザヘッドレス検出 `navigator.webdriver` 上書きは多くの場合効果薄**: 既にページ初期化後だと意味ない、起動時 flag で対応すべき
- **DOM クエリは querySelectorAll の index 依存がバグの温床**: 親の innerText を全部取る方が確実
- **拗音 (yo-on) は greedy 2-char lookup**: `きゃ → kya` のような 2 文字組み合わせを優先マッチングする greedy converter が必要、`き → ki` だけだと `ki + ya = kiya` になって不正解
- **連続ボーナスはスコアと session time の両方を伸ばす**: 7848 連続で +28 秒延長、+50% スコア倍率発動 = 隠れたメカニクス
- **Computer Use のブラウザ tier "read" 制約 vs agent-browser の synthetic dispatch**: Mac native の strict 制約を、自動化ヘッドレスブラウザの JS layer で迂回する設計選択肢
