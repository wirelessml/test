## 追補 (10:26〜11:05 JST) — Remote Control 接続失敗の RCA + claude-obsidian SessionStart hook 修正

### 経緯

10:26 の新セッション起動直後に `/remote-control` を実行 → 27.8 秒待たされた末に "Session creation failed" エラー。20 分後の 10:47 に再試行したら 0.8 秒で成功。なぜ 1 回目だけ失敗したかを systematic-debugging で根本原因まで追跡。

### 根本原因

claude-obsidian plugin v1.4.3 の `SessionStart:startup` hook が **prompt 型 hook を使用**しており、Claude Code 2.1.123 の内部バグを踏んでいた。SessionStart hook の `hook_non_blocking_error` attachment にエラーが記録されていた:

```
stderr: "Failed to run: ToolUseContext is required for prompt hooks. This is a bug."
exitCode: 1, durationMs: 1
hookName: SessionStart:startup
```

(`This is a bug.` は Claude Code 自身が「内部バグ」と認めている文言)

このエラー attachment が session の dirty state として残った状態で `/remote-control` を実行したため、Anthropic backend の session creation API で 27.8 秒の internal timeout 発動。20 分後は session が定常状態になり attachment 0 個で 0.8 秒で通った。

### 1 回目失敗 vs 2 回目成功 の決定的差分

| 指標 | 1 回目 (10:26:46) | 2 回目 (10:47:18) |
|---|---|---|
| session 起動からの経過 | 8.2 秒 | 約 20 分 |
| 直前 attachment 数 | 8 個 (約 64KB の hook 出力 + error) | 0 個 |
| transcript 累積 | 50.3 KB | 51.3 KB (+1KB のみ) |
| 結果 | 27.8 秒後に Session creation failed | 0.8 秒で成功 |

ペイロードサイズではなく **dirty hook state の有無**が決定要因。

### 対応 (両ファイル編集)

claude-obsidian の hooks.json から `SessionStart` 配列の **2 つ目 (`type: "prompt"`)** だけを削除。1 つ目の command 型 hook (`cat wiki/hot.md`) は機能上同じこと (hot.md の内容を context に注入) をするので残す → **機能損失ゼロで bug 回避**。

編集対象 (2 ファイル、cache と marketplace):
- `~/.claude/plugins/cache/claude-obsidian-marketplace/claude-obsidian/1.4.3/hooks/hooks.json`
- `~/.claude/plugins/marketplaces/claude-obsidian-marketplace/hooks/hooks.json`

バックアップ: `*.json.bak.20260430-110352` (両ディレクトリ)

検証 (verification-before-completion 準拠):
- ✓ JSON syntax (両ファイル `jq empty` OK)
- ✓ SessionStart 配列が 2 → 1 (command 型のみ残存)
- ✓ 他 hook (PostCompact / PostToolUse / Stop) 無傷
- ✓ cache ↔ marketplace 完全一致 (ドリフトなし)
- ✓ diff で削除したのは `type: "prompt"` の 4 行のみ

### 学んだこと

- **エラーメッセージの末尾 "This is a bug." を見逃さない**: Claude Code 内部バグ。Anthropic にレポート価値あり。
- **dirty state を含んだ session を export する API は危険**: Remote Control の session creation のような「session state を backend に送る」操作は、起動直後の不安定状態 (hook 完了直後) を避ける運用にする。応急対応としては「セッション起動から 30〜60 秒待ってから `/rc`」。
- **冗長な hook ペアは整理候補**: 同じ目的の hook が command + prompt の 2 種で重複していた。プラグイン側の冗長設計に気づけたのは bug のおかげ。
