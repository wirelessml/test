## 発生した問題と対応

### 問題 1: SSH セッションが Console session に届かない

**症状**: `Start-Process https://...` を SSH 経由で実行しても Chrome に新タブが開かない
**原因**: OpenSSH for Windows のセッション分離（SSH session ≠ Console session 1）
**対応**: ユーザー側で Chrome 手動起動 → bat ファイル経由で 4 URL 一括オープン
**学び**: SSH からの GUI 操作は本質的に困難。ファイル DL や config 編集は SSH で、GUI 操作はユーザー側で、と役割分担

### 問題 2: SuperWhisper Onboarding が設定をリセット

**症状**: SSH で preferences.json を編集して onboarding スキップ + showExperimentalModels=true 設定するも、SuperWhisper 起動時に巻き戻される
**原因**: SW 起動時にバリデーション/サニタイズが走り、不整合な設定を初期値に戻す
**対応**: GUI で onboarding を素直に通過 → モード設定は `default.json` で永続化される（こちらは生き残る）
**学び**: 一部のアプリは preferences の SSH 編集を許さない、その場合は GUI 操作が必須

### 問題 3: SuperWhisper Windows v1.3.9 に Scribe が不可視

**症状**: Voice Model dropdown に「Scribe」「ElevenLabs」表示なし、`showExperimentalModels=true` でも変化なし
**原因**: **Scribe は SuperWhisper Pro 限定機能**（`licenseType: empty` で Free tier 確定）
**対応**: 課金せず Python 自前路線で Scribe 直叩き構成 (`keybind_scribe.py`)
**学び**: 勝間 4/27 18:17 ツイ「SuperWhisper 買い切り、Scribe 課金なし」は勝間さん**自身が Lifetime 購入済 + BYO ElevenLabs API**を意味していた

## 学んだこと

- **勝間 X の likes 動態**: 本人主張ツイは 24h で 5-10 倍化け、返信ツイは横ばい。監視時にこの 2 種を区別する構造が必須

- **OpenSSH for Windows セッション分離**: GUI 操作（ブラウザ起動、アプリ最前面化）は SSH 経由で不可、ファイル/config/プロセスは可

### 真犯人: `StartOverride` という Windows の隠し最適化機構

**症状の組み合わせ**:
- BCD は完璧に正しい（`device partition=D:`、`osdevice partition=D:`、`path \Windows\system32\winload.efi`）
- `stornvme.sys` は実在（D:\Windows\System32\drivers\、311 KB）
- `stornvme` Service の Start = 0x0 (boot start)
- それでも 0x7B が出る

**原因**: `HKLM\OFFSYS\ControlSet001\Services\stornvme\StartOverride\0 = 0x3`

`StartOverride\<index>` は Windows の起動最適化機構で、**Start 値を boot 時にオーバーライドする**。Start=0x0 (boot start) と設定されていても、StartOverride\0=0x3 があれば実際の起動時には manual start (0x3) に降格される。Windows はインストール時や運用中に「このドライバは boot start で読み込まなくても問題ない」と自己判断したとき、勝手にこの値を書き込む。

旧 Plextor 環境では、Windows は SATA HDD → Plextor の NVMe SSD と移行していた可能性が高く、SATA 起動時に「NVMe ドライバは boot 時に不要」と判断して `stornvme\StartOverride\0=0x3` を書き込んでいた。これが新 Acer SSD への NVMe Direct クローン後の起動を完全に殺していた。

### 修正コマンド（後世のために）

```cmd
reg load HKLM\OFFSYS D:\Windows\System32\config\SYSTEM
reg delete "HKLM\OFFSYS\ControlSet001\Services\stornvme\StartOverride" /f
reg add "HKLM\OFFSYS\ControlSet001\Services\iaStorE" /v Start /t REG_DWORD /d 0x4 /f
reg unload HKLM\OFFSYS
```

`iaStorE` は .sys 不在のゴミ設定なので無効化（0x4=disabled）。

### 16:54〜17:14 起動経路の解明 + レジストリ恒常性確認

ユーザーから 4 つの根本的な疑問が連続で出てきたので、検証して回答。

#### Q1. 「なぜ 4/25 は外付け USB-C SSD で起動できて、4/29 は内蔵 NVMe クローンで起動できなかったのか」

**回答**: Windows のドライバ最適化挙動の違い。
- 4/25 SanDisk USB-C SSD 起動成功 → USB ストレージ経由なので Windows は `usbstor` ドライバで処理、`stornvme` の StartOverride 判定が走らなかった
- 4/29 内蔵 NVMe クローン → Windows は「現行構成で必須でないドライバ」として `stornvme` を `StartOverride=0x3 (Demand Start)` に降格 → ブート時に NVMe コントローラを認識できず 0xc0000001/0x7B
- 同じ SSD でもインタフェース層 (USB vs PCIe) が違うとドライバ評価ロジックが違う、という Windows 内部仕様の盲点

#### Q2. 「いま外付け USB-C SSD から起動するのは可能か」

**回答**: 不可能と判明。
- 現状のハードウェア構成 (Acer FA100 NVMe + Seagate SATA D:) では Windows が NVMe を essential とマーク済み
- 外付け USB-C SSD ブートを試みても、内蔵 NVMe を見て StartOverride 判定が走るため同じ症状が再発する可能性が高い
- = 4/25 の起動成功は「内蔵 SSD 死亡で Windows が SATA を essential として認識した状態」という特殊条件下だった

#### Q3. 「`stornvme\StartOverride\0` を削除した修正は再起動で消えないか?」

**回答**: 永続。
- StartOverride は Windows の自動 demote 結果のキャッシュ → 削除すれば「demote 履歴なし = essential」として扱われる
- 再起動しても `Start=0x0 (Boot Start)` のまま、StartOverride 復活なし
- ただし Insider build / Feature Update (25H2→26H1) / ハードウェア構成変化 でドライバ再評価が走る可能性は残る (低確率)

#### Q4. 「最終的に健康状態は問題ないか」

**回答**: 全項目クリア。検証コマンドで以下を確認:
