Global Admin なのに Copilot Studio で「追加のアクセスが必要です」が出るときの解決法

問題の概要

Microsoft 365 の Global Administrator（グローバル管理者）でログインしているにもかかわらず、以下のサービスでアクセス拒否が発生する。

よくある症状
  • Copilot Studio: 「追加のアクセスが必要です」画面で停止
  • Power Platform 管理センター: セキュリティロールのメンバー追加で権限エラー
  • Dynamics 365: ユーザーレコードが読み取り専用、「ロールの管理」ボタン非表示
  • Dataverse Web API: 403 prvAssignRole エラー

根本原因

Dataverse 環境が自動プロビジョニングされた際に、Global Admin に「システム管理者」セキュリティロールが自動付与されない場合がある（プロビジョニングバグ）。

チキン＆エッグ問題

Dataverse のセキュリティロールを割り当てるには prvAssignRole 権限が必要だが、この権限は「システム管理者」ロールにしか含まれていない。環境内に人間のシステム管理者が一人もいない場合、通常の方法ではロールを割り当てることが不可能になる。

Azure AD のロール（Global Admin 等）と Dataverse のセキュリティロール（システム管理者等）は完全に別の権限体系。Global Admin であっても Dataverse 内では権限がない場合がある。

解決手順

レガシー Dynamics 365 Web Client の「管理者に昇格」機能を使用する。この機能はサーバー側で Azure AD の Global Admin 権限を直接チェックし、Dataverse の prvAssignRole をバイパスする唯一の方法。

前提条件

  • Azure AD で Global Administrator または Power Platform Administrator ロールを持っていること
  • 対象ユーザーが Dataverse 環境に同期済みであること（ユーザーレコードが存在すること）

GUI 手順

1

レガシー設定画面にアクセス

ブラウザで以下の URL にアクセスする（{org} は環境の組織名に置き換え）:

https://{org}.crm7.dynamics.com/main.aspx?settingsonly=true

例: https://org996569cf.crm7.dynamics.com/main.aspx?settingsonly=true

2

セキュリティ → ユーザー に移動

設定 → システム → セキュリティ → ユーザー
3

ユーザーを選択

一覧から対象ユーザーのチェックボックスを選択する。ユーザー一覧が空の場合はビューを「有効なユーザー」に切り替えるか、プログラム的手法（後述）を使用する。

4

「管理者に昇格」ボタンをクリック

ツールバー（リボン）に表示される「管理者に昇格」ボタンをクリック。確認ダイアログで「OK」を押す。

成功すると、対象ユーザーに「システム管理者」セキュリティロールが即座に付与される。ページの再読み込みやサインアウトは不要。

プログラムからの実行

ユーザー一覧が空の場合（権限不足で表示されない場合）、ブラウザの開発者コンソールまたは自動化ツールから JavaScript で直接呼び出せる。

方法 A: JavaScript 関数の直接呼び出し

レガシー設定画面（main.aspx?settingsonly=true）のセキュリティ → ユーザー ページで、ブラウザの開発者コンソール (F12) を開き、以下を実行:

// {userId} は対象ユーザーの Dataverse systemuserid (GUID)
Mscrm.SystemUserActions.promoteToAdmin(["{userId}"]);

確認ダイアログが表示されるので「OK」をクリック。

方法 B: ASPXダイアログの直接アクセス

以下の URL をブラウザで直接開く（認証済みセッション内で）:

https://{org}.crm7.dynamics.com/_grid/cmds/dlg_promotetoadmin.aspx?userid={userId}

「選択したユーザーを管理者に昇格します。」と表示されるので「OK」をクリック。

方法 C: Playwright 等の自動化ツールから

// 1. レガシー設定画面に移動
await page.goto('https://{org}.crm7.dynamics.com/main.aspx?settingsonly=true');
// 2. セキュリティ → ユーザー に移動（略）
// 3. JavaScript で直接呼び出し
await page.evaluate(() => {
  Mscrm.SystemUserActions.promoteToAdmin(["{userId}"]);
});
// 4. 新しいタブで開くダイアログで OK をクリック
// （ダイアログは新しいタブとして開く場合がある）

確認方法

Dataverse Web API でロール割り当てを確認:

GET /api/data/v9.2/systemusers({userId})/systemuserroles_association?$select=name

レスポンスに "name": "システム管理者" が含まれていれば成功。

技術的な背景

dlg_promotetoadmin.aspx の仕組み

内部動作
  1. クライアント側: Mscrm.SystemUserActions.promoteToAdmin() が /_grid/cmds/dlg_promotetoadmin.aspx?userid={id} をダイアログとして開く
  2. サーバー側: ASPXページがAzure AD のディレクトリロール（Global Admin / Power Platform Admin）を直接チェック
  3. 認証バイパス: Dataverse の prvAssignRole 権限チェックをスキップして、直接 systemuserroles_association にレコードを挿入
  4. 結果: 対象ユーザーに「システム管理者」ロールが付与される

なぜこの問題が発生するのか

Dataverse のデフォルト環境が自動プロビジョニングされる際、通常は Global Admin に自動的にシステム管理者ロールが付与される。しかし以下のケースでは同期が発生しないことがある:

  • テナント作成後、しばらく経ってから Dataverse が追加された場合
  • Power Platform 管理センターから手動で Dataverse を有効化した場合
  • 環境のリージョンが特定の地域（日本等）の場合
  • タイミングの問題（Azure AD 同期ジョブの遅延・失敗）
