# 勝間和代の Scribe マニフェストから 32 時間 — 「漢数字問題」を Windows で追体験したら消えていた

2026 年 4 月 29 日午前 9 時 19 分。コワーキングスペースで、いま検証が終わった。

勝間和代さんが 4 月 28 日午前 1 時 15 分に投稿した「Scribe の最大の欠点」が、私の Windows 環境では再現しなかった。

---

## きっかけ

4 月 27 日 11 時 46 分、勝間さんが投稿した 1,168 likes のツイートで voice stack の世界が動いた。

> 「あんなに苦労して OpenAI の Whisper モデルと LLM の組み合わせを試していたんだけれども、Scribe の新しいバージョンの音声認識ソフトが優秀すぎて、LLM いらずで十分に認識してくれます」

ノイズ耐性、相槌の認識、ハルシネーションの解消。それまでの voice stack は Whisper + Groq Llama 70B での後処理が必須だった。それが ElevenLabs の Scribe 単体で要らなくなった、という宣言だった。

私はそれを観察ログに記録しながら、自分でも追体験したくなった。**真似するためではなく、勝間さんの観察と私の観察の差分を見るため**だ。

## 4 月 28 日朝、勝間さんの「欠点」報告

翌朝、勝間さんは Scribe の唯一の欠点を投稿した。

> 「Scribe の最大の欠点の一つが、過剰な正規化で数字を全部漢数字に直したがるので、その部分については後処理で LLM で補正するようにした方が良い」（4/28 01:56、20 likes）
>
> 「そこだけ LLM で直してます」（4/28 13:21、13 likes）

つまり「2026」と話しても「二千二十六」と書く。年号や金額や型番が漢数字で出る。これを軽い LLM プロンプトで補正する、という運用報告だった。

私はこれを受けて、ローカル正規表現で漢数字 → 算用数字を変換するスクリプトを書いた。LLM 不要、ネットワーク不要、決定的変換。8 ケースのユニットテストで全合格させて、Windows に転送した。

これで勝間さんの V4 構成（Scribe + Python トリム + 漢数字補正）が再現できる、はずだった。

## 4 月 29 日朝、Windows での実装

コワーキングに 6 時から SSH で接続して、HP ProBook（マウスコンピューター系の Windows 機）に Python 環境と 12 ファイルを配置した。Mac の `~/.zshrc` にある ELEVENLABS_API_KEY（51 文字、starter tier）を SSH 経由で Windows の HKCU\Environment に永続化した。

7 時前に物理着席して、SuperWhisper Windows v1.3.9 をインストール。

ここで詰まった。Scribe が出てこない。

Voice Model のドロップダウンには S1-Voice、Ultra、Nova 3 / 2 / Medical、Parakeet Multilanguage しかない。「Show experimental models」を ON にしても変わらない。

公式 Changelog を読み直して気づいた。Scribe は **Pro 限定機能**だった。勝間さんが 4/27 18:17 に投稿した「SuperWhisper 買い切り、Scribe 課金なし」の意味も、Pro Lifetime ライセンスを買って自前 ElevenLabs API キーで運用していることだとようやく腑に落ちた。

¥40,000（$249.99 / 1 ドル 159.68 円）。為替リスク込みで Lifetime を即決した。30 ヶ月で月額分岐。長期運用なら最安。

License Key を入れた瞬間、Voice Model のドロップダウンに **Scribe (NEW)** が現れた。‖ という ElevenLabs のロゴ付きで。

## マイクテスト、午前 9 時 19 分

メモ帳を開いて、左 Alt を押しながら言ってみた。

> 「これはテストです。コワーキングスペースでは音は出せません。**にせん にじゅうろくねん しがつ にじゅうくにち ごぜん くじ じゅうきゅうふん**。**ひゃくにじゅうビーのモデルを使用しています**」

数字は全部、日本語の数読みで発音した。勝間さんの 4/28 マニフェストが正しければ、漢数字で出てくるはずだ。

Ctrl+V で貼り付けた結果がこれだった。

> これはテストです。コワーキングスペースでは音は出せません。**2026 年 4 月 29 日午前 9 時 19 分**。**120B のモデル**を使用しています。

漢数字が、ない。

すべて算用数字。年月日も時刻もモデルサイズも、私が日本語読みで発話した数字が全部 Arabic numerals で出てきた。句読点も自動で適切に挿入された。ハルシネーションもなかった。

勝間さんの 4/28 マニフェストの「Scribe の最大の欠点」が、私の Windows 環境では再現しない。

## 32 時間で何が変わったのか

考えられる仮説は 3 つ。

**A. Scribe v2 の更新で漢数字問題が修正された**

ElevenLabs Scribe は 2026 年 3 月 12 日に v2 が GA リリースされている。SuperWhisper macOS では v2.11.0 で v2 に切り替わった。Windows v1.3.9 もバックエンド側は v2 を使っている可能性が高い。勝間さんがテストしたのが v1 だったかもしれないし、v2 移行直後で挙動が違ったのかもしれない。

**B. SuperWhisper 側に算用数字正規化レイヤーが追加された**

Scribe の生出力は漢数字でも、SuperWhisper の表示層で変換しているかもしれない。アプリ層での後処理は他の文字起こしツールでも一般的だ。

**C. 勝間さんの発話の癖と私の発話の癖が違う**

「にせん にじゅうろく」と「にせんにじゅうろく」では切れ目の入れ方が違う。Scribe のコンテキスト推論が私の発話には働いたが、勝間さんの発話には漢数字判定が走った可能性。これは可能性として薄いが完全に排除はできない。

最も濃厚なのは A だと思う。32 時間で勝間さんが観察した「欠点」が、ソフトウェアの更新で消えていた、というのが一番つじつまが合う。

## 観察哲学が見つけた差分

私が今日やった作業は、表面的には「勝間和代の voice stack を Windows で物理コピーする」だった。でも実際にやってみると、勝間さんが書いた前提条件が成立しない場面に出会う。

漢数字補正用に書いたローカル正規表現スクリプトは、現在の私の環境では発動する場面がない。**勝間さんの前提が古くなっていたから**だ。

でもこれは無駄ではない。edge case で漢数字が出る可能性は残っているし、Scribe v3 が出たときに退化するかもしれない。何より、**勝間さんが観察した瞬間と、私が追体験した瞬間で、ソフトウェアの状態が違っていた**という事実そのものが面白い。

「真似」だったら、勝間さんと同じ症状が出ないと「失敗」だと判断するだろう。でも追体験は違う。差分が出るからこそ、自分が観察者として何かを学ぶ。

勝間さんが 4 月 27 日に Scribe を発見してから 4 月 28 日に欠点を報告するまでの間、私が Mac から SSH で Windows に Python 環境を構築する間、ElevenLabs が静かに Scribe を更新していた。誰も騒がなかった。私が今朝発話するまで、Windows 環境でこの差分は誰の目にも触れていなかった。

技術的にはちっぽけな観察かもしれない。でも 32 時間という時間軸の中で、勝間さんと私の観察記録が積み重なって、ソフトウェアの進化の痕跡が浮き上がる。これが面白い。

## 残った課題

漢数字問題が消えたといっても、Scribe で完璧というわけではない。今日のテストでは「コワーキングスペースでは音は出せません」と私が小声で発話したのが正確に認識された。これは勝間さんの「ノイズ耐性が高い、小声でも認識する」という観察と一致した。

ただし環境が変わった時に挙動が同じとは限らない。雑踏、外国語混じり、専門用語の連続、辞書未登録の固有名詞。これから観察を続ける。

そして、勝間和代の voice stack 観察記は、これからも続く。
