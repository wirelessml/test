# なぜAIはPythonで作られるのか

ソース: [@pajostm on X](https://x.com/pajostm/status/2041961500290556362)（2026年4月）

## 1. Pythonは遅い — でも問題ない

Pythonは純粋な計算ではC/C++/Rustと比較して圧倒的に遅い。しかしAI開発でPythonが選ばれる理由は、Python自体が重い計算をしていないから。

## 2. Pythonはマネージャー、C++/CUDAはワーカー

- PyTorchのコアエンジン → C++
- CUDAカーネル → C
- NumPyの配列演算 → 裏側はC
- Pythonは指示を出すだけ。実行時間の99%はPythonの外で走っている

このパターン = 「低レベル計算の上に高レベルラッパーをかける」

## 3. 実行速度 vs 実験速度

- C++で全部書けば速いが、パラメータ調整のたびに再コンパイルが必要
- Pythonなら1行変えて数秒で再実行
- 研究段階では「イテレーション速度」が「実行速度」に勝つ
- Google、Meta、OpenAIの研究者は素早く実験を回す必要がある

## 4. エコシステムが決定打

NumPy、Pandas、Matplotlib、Scikit-learn、PyTorch、TensorFlow、Hugging Face、LangChainなど数千のライブラリ。

C++を主要言語にしていたら、すべてのチームがこれらを再構築することになる。エコシステムのロックインは本物。

## 5. プロダクションでは別の話

- 研究・訓練: Python
- 本番展開: C++/Rust等の高速言語
  - Meta: カスタムC++推論エンジン
  - Google: XLA
  - NVIDIA: TensorRT

ワークフロー: Pythonで研究・訓練 → 最適化された言語で展開

Claudeを動かすインフラも「苛烈な効率」のために構築されている。

## まとめ

Pythonが遅いのは事実。だが問題にならない理由:
1. 実際の計算はC++/CUDAが行う
2. 研究のイテレーション速度が最重要
3. エコシステムが圧倒的
4. プロダクションでは高速言語に移行する

「Python = 指揮者、C++/CUDA = 演奏者」

## 補足: 実例 — Rust→TypeScript移植をOpusに一任

[@bcherny on X](https://x.com/bcherny/status/2042352725481177315) の事例:

- Rust+NAPIファイルインデクサをネイティブTypeScriptに移植
- プロンプト:「コードを移植し、テストスイートを完璧に通過させ、プロファイリングし、PRを提出せよ。新しい実装が速いことを決定的に証明するまで止めるな」
- 結果: NAPIのFFIオーバーヘッドがボトルネックだったため、TSの方が速くなった
- さらに「p99 < 10ms」「スタッタリング修正」まで追い込んで完了

→ 「速い言語で書けば速い」とは限らない。FFI/ラッパーのオーバーヘッドが律速になることがある。

## 補足: ハーネスの質がアウトプットを変える

素のClaude Code CLIより、GitHub Copilot CLIでClaude（Opus）を選んで使う方がアウトプットの品質が高いという観察がある。

同じClaude Opusエンジンでも、ハーネス（指揮者）の質でアウトプットが変わる:
- Copilot CLIのシステムプロンプトがGitHubのコードベース知識で強化されている
- コンテキスト管理が最適化されている
- ツール呼び出しの制御が洗練されている

→ Python×AIと同じ構造。エンジンが同じでも、ラッパーの設計で結果が変わる。

## 結論

「PythonはAIを動かしているわけではない。PythonはAIを統括しているのだ。」

人々が心配する遅さは、Pythonが決して触れない層に存在する。Pythonが「このモデルを訓練せよ」と言う頃には、すでにその下で数千ものC++やCUDAの操作が発火している。

AI業界は、まず人間の速度を最適化し、次に機械の速度を最適化した。その決定が、今日のAI業界の全てを形作った。
